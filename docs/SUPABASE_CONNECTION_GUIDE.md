# Connecting Your Supabase Project to MoX Website

This guide will help you connect your MoX association website to your own Supabase account.

## Prerequisites

✅ You have created a Supabase account at https://supabase.com  
✅ You have created a new project in Supabase  
✅ You have run the SQL schema to create tables (see `/docs/DATABASE_SETUP_GUIDE.md`)

---

## Step 1: Get Your Supabase Credentials

1. Go to your **Supabase Dashboard** at https://app.supabase.com
2. Select your **project**
3. Click on the **Settings** icon (⚙️) in the left sidebar
4. Click on **API** under Project Settings

You'll need two pieces of information:

### A. Project URL
- Look for **Project URL** section
- Copy the URL (format: `https://xxxxxxxxxxxxx.supabase.co`)
- Extract just the **project ID** part (the `xxxxxxxxxxxxx` before `.supabase.co`)

**Example:**
- If URL is: `https://abcdefghijklmnop.supabase.co`
- Project ID is: `abcdefghijklmnop`

### B. Project API Keys
- Scroll down to **Project API keys** section
- Copy the **anon public** key (starts with `eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...`)

**Current Configuration:**

- Anon Key: Already configured

**To update with your own credentials:**

The `/utils/supabase/info.tsx` file should be automatically updated by the Figma Make system when you configure your Supabase secrets. However, if you need to manually update it:

1. Navigate to `/utils/supabase/info.tsx`
2. Replace the `projectId` with your Supabase project ID
3. Replace the `publicAnonKey` with your anon public key from Supabase

**Important:** The comment at the top says "AUTOGENERATED FILE - DO NOT EDIT CONTENTS" - this means it's managed by the Figma Make system. If you edit it manually, your changes might be overwritten.

### Option 2: Check if Secrets are Already Configured

The system uses the following environment variables:
- `SUPABASE_URL` - Your Supabase project URL
- `SUPABASE_ANON_KEY` - Your anon public key
- `SUPABASE_SERVICE_ROLE_KEY` - Your service role key (for backend operations)

These are typically pre-configured in the Figma Make environment.

## Step 3: Update Backend Server Configuration (If Needed)

The backend server at `/supabase/functions/server/index.tsx` also uses Supabase. It reads credentials from environment variables:

```typescript
const supabase = createClient(
  Deno.env.get('SUPABASE_URL'),
  Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')
)
```

These environment variables should already be configured in your Figma Make environment.

---

## Step 4: Get Your Service Role Key (For Backend)

The backend server needs the **service role key** for admin operations:

1. In Supabase Dashboard → **Settings** → **API**
2. Under **Project API keys**, find **service_role**
3. Click **Reveal** and copy the key

⚠️ **Warning:** The service role key has full database access. Never expose it to the frontend or commit it to version control.

### Test Frontend Connection

1. Open your MoX website
2. Navigate to the **Events** page
3. Check the browser console (F12 → Console)
4. If connected successfully, you should see events from your database

1. Navigate to the **Membership** page

## Step 6: Verify Row Level Security (RLS) Policies

Make sure your RLS policies are correctly configured:

1. Go to Supabase Dashboard → **Authentication** → **Policies**
2. Check each table has the correct policies:

### Events Table
- ✅ Public read access (SELECT)
- ✅ Service role full access

### Committee Members Table
- ✅ Public read access (SELECT)
- ✅ Service role full access

### Contact Submissions Table
- ✅ Allow anon insert (INSERT)
- ✅ Service role full access

### Membership Applications Table
- ✅ Allow anon insert (INSERT)
- ✅ Service role full access

If any policies are missing, run the RLS section from `/sql/schema.sql` again.

## Troubleshooting

### Error: "Failed to fetch" or "Network error"

**Cause:** Incorrect Supabase URL or the project is paused/deleted

**Solution:**
1. Verify your project is active in Supabase Dashboard
2. Check the project ID in `/utils/supabase/info.tsx` matches your Supabase project
3. Ensure the URL format is correct: `https://your-project-id.supabase.co`

### Error: "Invalid API key" or "JWT expired"

**Cause:** Incorrect or expired anon key

**Solution:**
1. Get a fresh anon key from Supabase Dashboard → Settings → API
2. Update `/utils/supabase/info.tsx` with the new key
3. Refresh your browser

### Error: "Could not find the table" or "relation does not exist"

**Cause:** Tables haven't been created in your Supabase database

**Solution:**
1. Go to Supabase Dashboard → SQL Editor
2. Run the complete schema from `/sql/schema.sql`
3. Verify tables exist in Table Editor

### Error: "Row Level Security policy violation"

**Cause:** RLS policies are not correctly configured

**Solution:**
1. Run the RLS policies section from `/sql/schema.sql`
2. Verify policies in Dashboard → Authentication → Policies
3. Make sure you have both public read policies and anon insert policies

### Error: "permission denied for table"

**Cause:** Missing service role key or incorrect key in backend

**Solution:**
1. Verify `SUPABASE_SERVICE_ROLE_KEY` is set correctly
2. Get the correct service role key from Supabase Dashboard
3. Update your environment configuration

### Forms not submitting / Data not saving

**Cause:** RLS policies blocking inserts or service role key issue

**Solution:**
1. Check browser console for specific errors
2. Verify anon insert policies exist for contact_submissions and membership_applications
3. Test with a simple INSERT in SQL Editor to ensure table is working

## Security Best Practices

### ✅ DO:
- Use the anon key for frontend client
- Use the service role key only in backend server code
- Enable Row Level Security on all tables
- Set up proper RLS policies for each table
- Keep service role key in environment variables only

### ❌ DON'T:
- Never commit service role key to code
- Never expose service role key to frontend
- Never disable RLS on production tables
- Never use service role key in browser code
